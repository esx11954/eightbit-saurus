# テーブル設計

## テーブル設計とは

テーブル設計とは、システムで扱うデータをどのようなテーブル・列・制約で管理するかを決める作業です。  
設計の良し悪しがシステムの保守性・パフォーマンス・データ整合性に直結するため、実装前に十分な検討が必要です。

---

## Oracleの主なデータ型

列を定義する際は、格納するデータの性質に合ったデータ型を選択します。

### 数値型

| データ型 | 説明 | 例 |
|---|---|---|
| `NUMBER(p)` | 整数（桁数p） | `NUMBER(4)` → 最大9999 |
| `NUMBER(p, s)` | 小数（全体桁数p、小数桁数s） | `NUMBER(6, 2)` → 最大9999.99 |

### 文字列型

| データ型 | 説明 |
|---|---|
| `VARCHAR2(n)` | 可変長文字列（最大n文字）。余った分は領域を消費しない |
| `CHAR(n)` | 固定長文字列（常にn文字分の領域を確保）。不足分は空白で埋まる |
| `CLOB` | 大量のテキストデータ（最大128TB） |

:::tip
一般的な文字列列には `VARCHAR2` を使用します。  
`CHAR` は郵便番号など**常に同じ桁数**が確定している列に向いています。

また、日本語などのマルチバイト文字を格納する場合は `VARCHAR2(n CHAR)` のように  
`CHAR` セマンティクスを指定すると、文字数単位でサイズを指定できます。
:::

### 日付・時刻型

| データ型 | 説明 |
|---|---|
| `DATE` | 日付と時刻（年月日・時分秒）を格納する |
| `TIMESTAMP` | DATEより高精度（ミリ秒以下まで格納可能） |

```sql
-- 現在日時を挿入する
INSERT INTO orders (order_id, order_date) VALUES (1, SYSDATE);
```

## 正規化

**正規化**とは、データの冗長性（重複）を排除し、データの整合性を保ちやすい形にテーブルを整理するプロセスです。

### 正規化前（非正規形）の問題点

以下のような1テーブルにすべてのデータを詰め込んだ設計は問題があります。

| 注文ID | 顧客名 | 顧客住所 | 商品名1 | 数量1 | 商品名2 | 数量2 |
|---|---|---|---|---|---|---|
| 1 | 田中太郎 | 東京都 | 商品A | 2 | 商品B | 1 |
| 2 | 山田花子 | 大阪府 | 商品A | 3 | | |

- 顧客情報が注文のたびに重複して格納される
- 商品が増えると列を追加する必要がある
- 顧客の住所が変更になった場合、全ての行を更新する必要がある

### 第1正規形（1NF）

**繰り返し項目を排除し、1つのセルに1つの値のみ格納する**

| 注文ID | 顧客名 | 顧客住所 | 商品名 | 数量 |
|---|---|---|---|---|
| 1 | 田中太郎 | 東京都 | 商品A | 2 |
| 1 | 田中太郎 | 東京都 | 商品B | 1 |
| 2 | 山田花子 | 大阪府 | 商品A | 3 |

### 第2正規形（2NF）

**主キーの一部だけに依存する列（部分関数従属）を別テーブルに分離する**

主キーが「注文ID + 商品名」の複合キーの場合、「顧客名」「顧客住所」は注文IDだけに依存しています。  
この部分を別テーブルに分離します。

**注文テーブル**

| 注文ID | 顧客ID | 商品名 | 数量 |
|---|---|---|---|
| 1 | 101 | 商品A | 2 |
| 1 | 101 | 商品B | 1 |
| 2 | 102 | 商品A | 3 |

**顧客テーブル**

| 顧客ID | 顧客名 | 顧客住所 |
|---|---|---|
| 101 | 田中太郎 | 東京都 |
| 102 | 山田花子 | 大阪府 |

### 第3正規形（3NF）

**主キー以外の列に依存する列（推移関数従属）を別テーブルに分離する**

例えば注文テーブルに「担当者ID」と「担当者名」があり、担当者名が担当者IDに依存している場合、  
担当者情報を別テーブルに分離します。

## ER図

**ER図（Entity-Relationship Diagram）** は、テーブル（エンティティ）間の関係を視覚的に表した図です。  
設計の全体像を把握・共有するために作成します。

### 主な表記要素

| 要素 | 説明 |
|---|---|
| エンティティ（四角） | テーブルを表す |
| 属性（楕円 or 列一覧） | テーブルの列を表す |
| リレーションシップ（線） | テーブル間の関係を表す |

### カーディナリティ（多重度）

テーブル間の「1対1」「1対多」「多対多」の関係を表します。

```
顧客 ─── 注文        1人の顧客が複数の注文を持つ（1対多）
注文 ─── 商品        1つの注文に複数の商品が含まれる（多対多 → 中間テーブルで解消）
```

多対多の関係は直接テーブルで表現できないため、**中間テーブル**（関連テーブル）を作成して解消します。

```
注文 ─── 注文明細 ─── 商品
```

## テーブル設計書

実際の開発現場では、以下のような項目をまとめた**テーブル設計書**を作成します。

| 項目 | 説明 |
|---|---|
| テーブル名（論理名） | 業務上の名称（例：顧客マスタ） |
| テーブル名（物理名） | DB上の名称（例：customers） |
| 列名（論理名） | 業務上の列の名称（例：顧客ID） |
| 列名（物理名） | DB上の列の名称（例：customer_id） |
| データ型 | 格納するデータの型（例：NUMBER(4)） |
| NOT NULL | NULL不可かどうか |
| 主キー / 外部キー | 制約の種類 |
| 備考 | 特記事項 |

## 設計時の主な考慮点

| 観点 | 内容 |
|---|---|
| 命名規則 | テーブル名・列名に一貫したルールを設ける（スネークケース等） |
| データ型の選定 | 格納データに合った型を選ぶ。過大なサイズ定義は避ける |
| NULL許容の判断 | 必須項目には NOT NULL を設定する |
| 主キーの選定 | 業務上の意味を持たない連番（サロゲートキー）の使用も検討する |
| インデックスの計画 | 検索頻度の高い列をあらかじめ把握しておく |
| 正規化と非正規化のバランス | 過度な正規化はJOINが増えパフォーマンスに影響することもある |
