# インデックス

## インデックスとは

**インデックス（索引）** とは、テーブルの特定の列に対して作成する検索効率化のための仕組みです。  
本の巻末にある「索引」と同じ原理で、目的のデータの場所をすばやく特定できるようにします。

インデックスがない場合、DBは条件に合う行を見つけるためにテーブルの全行を順番に読み込む  
**フルテーブルスキャン** を行います。データ量が多いほどこの処理は遅くなります。

---

## インデックスの効果

`items` テーブルには約28,000件のデータが格納されています。  
以下のクエリでインデックスの有無による違いを体感できます。

```sql
-- インデックスなしで検索（フルテーブルスキャン）
SELECT * FROM items WHERE name = '商品_15000';
```

`name` 列にインデックスを作成した後に同じクエリを実行すると、検索速度の違いが確認できます。

---

## インデックスの作成

`CREATE INDEX` 文を使用してインデックスを作成します。

```sql
CREATE INDEX インデックス名 ON テーブル名(列名);
```

```sql
-- items テーブルの name 列にインデックスを作成する
CREATE INDEX idx_items_name ON items(name);
```

複数列を組み合わせた**複合インデックス**も作成できます。

```sql
-- name と type の複合インデックスを作成する
CREATE INDEX idx_items_name_type ON items(name, type);
```

---

## インデックスの確認

作成済みのインデックスはデータディクショナリビューで確認できます。

```sql
-- 自分が作成したインデックスの一覧を確認する
SELECT index_name, table_name, column_name
FROM user_ind_columns
WHERE table_name = 'ITEMS';
```

---

## インデックスの削除

```sql
DROP INDEX インデックス名;
```

```sql
DROP INDEX idx_items_name;
```

---

## インデックスが自動作成されるケース

OracleDBでは以下の制約を定義すると、自動的にインデックスが作成されます。

| 制約 | 動作 |
|---|---|
| PRIMARY KEY | 主キー列に自動でユニークインデックスが作成される |
| UNIQUE | 対象列に自動でユニークインデックスが作成される |

`items` テーブルの `item_id` には PRIMARY KEY が定義されているため、  
作成時点でインデックスが自動生成されています。

---

## インデックスを使用しない場合

インデックスを作成していても、クエリの書き方によっては使用されないことがあります。

```sql
-- インデックスが使われない例（列に関数を適用している）
SELECT * FROM items WHERE UPPER(name) = '商品_15000';

-- インデックスが使われる例
SELECT * FROM items WHERE name = '商品_15000';
```

```sql
-- インデックスが使われない例（LIKE の前方一致以外）
SELECT * FROM items WHERE name LIKE '%15000';

-- インデックスが使われる例（前方一致）
SELECT * FROM items WHERE name LIKE '商品_%';
```

---

## インデックスの注意点

インデックスは検索を高速化しますが、以下のデメリットも伴います。

| デメリット | 説明 |
|---|---|
| 更新コストの増加 | INSERT / UPDATE / DELETE のたびにインデックスも更新されるため、更新処理が遅くなる |
| ストレージ使用量の増加 | インデックスはテーブルとは別に領域を消費する |
| 効果がない場合がある | データの種類が少ない列（性別など）ではフルスキャンより遅くなることもある |

:::tip
インデックスは「頻繁に検索条件に使われる列」「データの種類が多い列」に作成すると効果的です。  
闇雲に作成せず、必要な箇所に絞って定義しましょう。
:::
