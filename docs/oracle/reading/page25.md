# バックアップ/リカバリ基礎

## なぜバックアップが必要か

データベースはハードウェア障害・オペレーションミス・ソフトウェアのバグなど様々な原因で  
データが失われるリスクがあります。バックアップを取得しておくことで、障害発生時にデータを  
復元（リカバリ）することが可能になります。

本番環境では「いつ障害が起きても復旧できる」状態を常に維持することが求められます。

---

## OracleDBの主な構成ファイル

バックアップ・リカバリを理解する上で、以下のファイルの役割を把握しておく必要があります。

| ファイル | 説明 |
|---|---|
| データファイル（.dbf） | テーブルなど実際のデータを格納するファイル |
| 制御ファイル（CONTROLFILE） | DBの構成情報（データファイルの場所・バックアップ履歴等）を管理する |
| REDOログファイル | コミット済みの変更履歴を記録する。障害時のリカバリに使用する |
| アーカイブログファイル | REDOログの退避ファイル。アーカイブログモード時に生成される |
| SPFILE（初期化パラメータファイル） | DBの動作設定（メモリサイズ・ファイルパス等）を管理するバイナリファイル |

---

## アーカイブログモード

OracleDBには以下の2つの動作モードがあります。

| モード | 説明 |
|---|---|
| NOARCHIVELOGモード | REDOログが上書きされるため、完全リカバリが不可能。開発環境向け |
| ARCHIVELOGモード | REDOログが退避（アーカイブ）されるため、障害直前までの完全リカバリが可能。本番環境向け |

```sql
-- 現在のモードを確認する
SELECT log_mode FROM v$database;

-- アーカイブログモードへの切り替え（要DB停止・再起動）
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE OPEN;
```

## バックアップの種類

### 物理バックアップと論理バックアップ

| 種類 | 説明 | 代表ツール |
|---|---|---|
| 物理バックアップ | データファイル等をファイル単位でそのままコピーする | RMAN |
| 論理バックアップ | データをSQL形式でエクスポートする | Data Pump（expdp/impdp） |

本番環境では主に**RMANによる物理バックアップ**が使用されます。

### 完全バックアップと増分バックアップ

| 種類 | 説明 |
|---|---|
| 完全バックアップ（Full） | データベース全体をバックアップする |
| 増分バックアップ（Incremental） | 前回バックアップ以降に変更されたブロックのみをバックアップする |

増分バックアップはバックアップ時間とストレージを節約できますが、  
リカバリ時は複数のバックアップを順に適用する必要があります。

---

## RMAN（Recovery Manager）

**RMAN** はOracleDBに付属するバックアップ・リカバリ専用ツールです。  
コマンドラインから操作し、バックアップの取得・管理・リストア・リカバリを一括して行えます。

### RMANへの接続

```bash
# ターゲットDBにSYSDBA権限で接続する
$ rman target /
```

### 主なRMANコマンド

| コマンド | 説明 |
|---|---|
| `BACKUP DATABASE` | データベース全体をバックアップする |
| `BACKUP ARCHIVELOG ALL` | 全アーカイブログをバックアップする |
| `LIST BACKUP` | バックアップ履歴を一覧表示する |
| `RESTORE DATABASE` | バックアップからデータファイルを復元する |
| `RECOVER DATABASE` | REDOログを適用してデータを最新状態に戻す |
| `RESTORE SPFILE` | SPFILEを復元する |
| `RESTORE CONTROLFILE` | CONTROLFILEを復元する |

```sql
-- データベース全体と全アーカイブログをバックアップする
RMAN> BACKUP DATABASE PLUS ARCHIVELOG;

-- バックアップ履歴を確認する
RMAN> LIST BACKUP;
```

---

## リストアとリカバリの違い

混同しやすい2つの概念を整理します。

| 操作 | 説明 |
|---|---|
| **リストア（RESTORE）** | バックアップからファイルを物理的にコピーして戻す作業 |
| **リカバリ（RECOVER）** | リストア後にREDOログ・アーカイブログを適用して、障害発生直前の状態まで戻す作業 |

```
バックアップ取得時点  →  アーカイブログ適用  →  障害発生直前
[    RESTORE     ]       [      RECOVER      ]
```

---

## リカバリの流れ

構成ファイルが失われた場合の一般的なリカバリ手順です。

```
1. ダミーインスタンスを起動してRMANに接続する
         ↓
2. バックアップセットからSPFILEをリストアする
         ↓
3. SPFILEを使ってインスタンスを起動する（マウントなし）
         ↓
4. バックアップセットからCONTROLFILEをリストアする
         ↓
5. DBをマウントし、データファイルをリストアする
         ↓
6. アーカイブログ・REDOログを適用してリカバリする
         ↓
7. DBをオープンしてデータを確認する
```

---

## 高速リカバリ領域（FRA）

**高速リカバリ領域（Fast Recovery Area）** は、バックアップ・アーカイブログ・REDOログの  
退避先として使用するディスク領域です。RMANのデフォルトの保管先になります。

```sql
-- 高速リカバリ領域のパスとサイズを確認する
SHOW PARAMETER recovery_file_dest;

-- パスとサイズを設定する
ALTER SYSTEM SET db_recovery_file_dest = '/u01/app/oracle/RMAN';
ALTER SYSTEM SET db_recovery_file_dest_size = 10G;
```

:::tip
高速リカバリ領域の容量が不足するとバックアップが失敗します。  
定期的に古いバックアップを削除し、空き容量を確保しましょう。

```sql
-- 不要なバックアップを削除する（保持ポリシーに基づく）
RMAN> DELETE OBSOLETE;
```
:::
